# Getting Started

This guide assumes you want to run `@flowindex/agent-wallet` as an MCP server.

## 1. Add the MCP Server

Claude Desktop example:

```json
{
  "mcpServers": {
    "flow-wallet": {
      "command": "npx",
      "args": ["@flowindex/agent-wallet"],
      "env": {
        "FLOW_NETWORK": "mainnet",
        "APPROVAL_REQUIRED": "true"
      }
    }
  }
}
```

If you are using another MCP host, the same environment variables apply. Only the place where you register the command changes.

## 2. Pick a Signer Mode

The server auto-detects the signer type from the environment:

| Signer Type | Trigger | Use Case |
|---|---|---|
| `cloud-interactive` | No key env vars and no `FLOWINDEX_TOKEN` | Best default for interactive use with a real user in the loop |
| `cloud` | `FLOWINDEX_TOKEN` is set | Headless cloud wallet access |
| `local-mnemonic` | `FLOW_MNEMONIC` is set | Local signing with an HD wallet |
| `local-key` | `FLOW_PRIVATE_KEY` is set | Local signing with a raw private key |

### Local Private Key

```bash
export FLOW_PRIVATE_KEY="your-64-char-hex-private-key"
export FLOW_ADDRESS="0x1234567890abcdef"
export FLOW_NETWORK="mainnet"
```

If you omit `FLOW_ADDRESS`, the server will try to discover accounts associated with the public key through FlowIndex.

### Mnemonic Phrase

```bash
export FLOW_MNEMONIC="word1 word2 word3 ... word12"
export FLOW_NETWORK="mainnet"
```

### Cloud Wallet

```bash
export FLOWINDEX_TOKEN="eyJhbG..."
export FLOW_NETWORK="mainnet"
```

### Interactive Cloud Login

```bash
export FLOW_NETWORK="mainnet"
```

With no local keys and no `FLOWINDEX_TOKEN`, the server starts in interactive mode. The AI agent calls `wallet_login`, gets a browser URL, and the user authenticates there.

## 3. Environment Variables

| Variable | Required | Default | Description |
|---|---|---|---|
| `FLOW_NETWORK` | No | `mainnet` | Flow network: `mainnet` or `testnet` |
| `FLOW_MNEMONIC` | No | -- | BIP-39 mnemonic phrase for local signing |
| `FLOW_PRIVATE_KEY` | No | -- | Hex-encoded private key for local signing |
| `FLOW_ADDRESS` | No | -- | Explicit Flow address; otherwise the server attempts discovery |
| `FLOW_KEY_INDEX` | No | `0` | Key index on the Flow account |
| `FLOW_SIG_ALGO` | No | `ECDSA_secp256k1` | `ECDSA_P256` or `ECDSA_secp256k1` |
| `FLOW_HASH_ALGO` | No | `SHA2_256` | `SHA2_256` or `SHA3_256` |
| `EVM_PRIVATE_KEY` | No | -- | Dedicated EVM private key; falls back to Flow key material when unset |
| `EVM_ACCOUNT_INDEX` | No | `0` | EVM derivation index when using a mnemonic |
| `FLOWINDEX_TOKEN` | No | -- | FlowIndex wallet token for cloud signing |
| `FLOWINDEX_URL` | No | `https://flowindex.io/api` | FlowIndex API base URL |
| `FLOW_SIMULATOR_ENABLED` | No | `true` | Enable mainnet preflight simulation calls |
| `FLOW_SIMULATOR_URL` | No | `https://simulator.flowindex.io/api` | Simulator base URL |
| `ALLOW_RAW_CADENCE_SIGNING` | No | `false` | Allow headless raw Cadence transactions to execute without the extra approval gate |
| `APPROVAL_REQUIRED` | No | `true` | Require manual confirmation before signing Cadence transactions |
| `ETHERSCAN_API_KEY` | No | -- | Optional Etherscan key for EVM ABI lookup |

## 4. Preflight Simulation

Mainnet transaction templates can be simulated before signing.

- `simulate_template` is the explicit read-only MCP tool
- `execute_template` also attempts a preflight simulation on mainnet
- if the simulator call fails, transaction execution still proceeds
- set `FLOW_SIMULATOR_ENABLED=false` to skip simulator calls entirely

The public FlowIndex simulator is a mainnet fork, so testnet requests are skipped with an explicit reason instead of failing the wallet flow.

## 5. Optional Cadence Authoring Add-On

If you want the agent to write and repair custom Cadence instead of only using wallet templates, add the Cadence MCP server alongside `agent-wallet`.

Claude Desktop example:

```json
{
  "mcpServers": {
    "flow-wallet": {
      "command": "npx",
      "args": ["@flowindex/agent-wallet"]
    },
    "cadence": {
      "command": "npx",
      "args": ["-y", "mcp-remote", "https://cadence-mcp.up.railway.app/mcp"]
    }
  }
}
```

That gives the agent:

- Flow wallet signing and approval through `agent-wallet`
- documentation lookup through Cadence MCP
- Cadence checking, symbol lookup, and security scanning through Cadence MCP
- freeform read-only execution through `execute_cadence_script`
- raw transaction preview through `simulate_cadence_transaction`

If you are using a coding-focused agent such as Codex or Claude Code, you can also add the Cadence skill:

```bash
npx skills add outblock/cadence-lang.org
```

The skill is optional. It improves long-lived context and language recall, but the signing path should still work with `agent-wallet` alone.

Today the advanced freeform path is:

- run raw scripts with `execute_cadence_script`
- preview raw transactions with `simulate_cadence_transaction`
- submit raw transactions with `execute_cadence_transaction`
- keep in mind that headless raw Cadence signing stays approval-gated unless `ALLOW_RAW_CADENCE_SIGNING=true`

## 6. Approval Flow

`APPROVAL_REQUIRED=true` is the default.

- `execute_template` proposes a transaction
- if approval is required, the server returns a pending transaction ID
- `confirm_transaction` signs and submits it later
- `cancel_transaction` discards it
- `list_pending` shows everything still waiting

This keeps the MCP tool interface safe by default while still allowing fully headless mode when you set `APPROVAL_REQUIRED=false`.

## 7. Local Runtime And Inspector

If you want to run the server directly outside an MCP host:

```bash
npx @flowindex/agent-wallet
```

To inspect the MCP tool surface locally:

```bash
npx @modelcontextprotocol/inspector node node_modules/@flowindex/agent-wallet/dist/index.js
```

## 8. Cadence Templates

The server ships with 70+ built-in Cadence templates across seven categories:

| Category | Count | Description |
|----------|-------|-------------|
| `base` | 9 | Account info, storage, keys, child accounts, surge factor |
| `token` | 3 | Fungible token transfers and vault setup |
| `collection` | 5 | NFT transfers (single, batch, NBA Top Shot) |
| `evm` | 9 | COA creation, EVM calls, cross-VM flow operations |
| `bridge` | 14 | FlowEVMBridge token and NFT bridging (both directions) |
| `hybrid-custody` | 22 | Child account FT/NFT operations, bridging, and queries |
| `lost-and-found` | 7 | Unclaimed token/NFT queries, claims, and safe sends |

Use `list_templates` to browse available templates and `get_template` to inspect a specific template's source code and argument schema.
